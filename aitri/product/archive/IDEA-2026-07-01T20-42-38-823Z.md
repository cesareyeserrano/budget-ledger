# Project Idea — Ledger (T-Ledger)

> Seed brief. La definición detallada del producto vive en `aitri/product/idea_context/`
> (Especificación Técnica Completa, mockups, stack, notas de dominio) y es **autoritativa**.
> Este archivo captura los inputs de tierra-verdad confirmados con el usuario.
> Actualizado 2026-07-01 con correcciones: es una **app web** (no app móvil, no mobile-first);
> en v1 el móvil (responsive compacto) muestra **solo** el módulo de registro de movimientos;
> Presupuesto y Dashboard son de **escritorio**. Se elimina la distribución proporcional.
> El borrado reasigna a una categoría fija **"Sin asignar"** por grupo.

---

## Problem

Las personas que quieren llevar sus finanzas personales contra un presupuesto no tienen una
herramienta que combine captura rápida de movimientos con planeación y análisis anual sobre los
mismos datos. Las apps existentes son demasiado complejas, caras, o separan la captura del
presupuesto. Ledger es una **app web** que resuelve registrar movimientos de dinero contra un
presupuesto mensual/anual, con una taxonomía de categorías que el propio usuario gestiona. La
misma app web es responsive: en pantalla pequeña (móvil, v1) muestra una versión **compacta** con
solo el módulo de captura; en pantalla grande (escritorio) es la app completa (presupuesto + dashboard).

## Target Users

Usuario individual que gestiona sus finanzas personales (single-user en v1). Cómodo con herramientas
web básicas. Usa la app web desde el navegador: en el escritorio para planear/analizar el presupuesto
anual, y en pantalla pequeña (responsive) para capturar movimientos rápido.
También es un proyecto de portafolio técnico del autor (César Augusto).

## Current Pain / Baseline

Hoy se resuelve con hojas de cálculo o apps genéricas: la captura y el presupuesto viven separados,
la taxonomía es rígida, y no hay una vista de plan-vs-realidad de 12 meses con roll-ups jerárquicos.
Baseline: prototipo HTML offline de alta fidelidad ya construido (`idea_context/Ledger (offline).html`)
cuyas reglas de cálculo (§4 de la spec) son la implementación de referencia.

## Business Rules

The system must permitir registrar un movimiento (tipo, categoría, subcategoría opcional, mes, monto) y sumarlo al Ejecutado de la hoja destino. La entrada de monto es un input estándar de la app web (sin teclado numérico ad-hoc).
The system must gestionar una jerarquía de 3 niveles editables (Grupo → Categoría → Subcategoría) bajo 3 tipos fijos (Gasto/Ingreso/Transferencia), con CRUD completo.
The system must preservar los movimientos al borrar una categoría, reasignándolos a una categoría fija "Sin asignar" dentro del mismo grupo (sin huérfanos). La categoría "Sin asignar" es una por grupo, no editable ni borrable, y solo se muestra cuando tiene datos.
The system must calcular roll-ups de abajo hacia arriba: subcategoría → categoría → grupo → tipo, para Presupuestado y Ejecutado.
The system must mostrar una grilla de presupuesto de 12 meses en escritorio (columnas Presupuestado/Ejecutado por mes, edición en línea de las hojas, sticky). El presupuesto es un módulo de escritorio.
The system must derivar signo, color y regla de varianza por tipo (Gasto sobre-presupuesto = error; Ingreso favorable/bajo; Transferencia neutra).
The system must mostrar un dashboard con los 7 indicadores mínimos (ingresos, gastos, balance neto, tasa de ahorro, adherencia, top categorías, sobre-presupuesto) respetando el filtro Mes/Año. El dashboard es un módulo de escritorio.
The system must ser una sola app web responsive; en pantalla pequeña (móvil, v1) muestra la versión compacta con SOLO el módulo de registro de movimientos, y en escritorio la app completa (registro + presupuesto + dashboard), sobre los mismos datos y reglas.
The system must persistir los datos en localStorage (jerarquía, budgets, actuals, movements) en v1.
The system must implementar el sistema de diseño César Augusto (tema oscuro único, tipografía mono, bordes sobre rellenos, sin emoji/gradientes) con los tokens exactos de la spec.

## Success Criteria

Given un usuario nuevo, when abre la app por primera vez, then ve datos semilla coherentes (jerarquía + montos dummy) y puede operar sin configuración.
Given el módulo de registro (en móvil compacto o en escritorio), when elige tipo + categoría + mes + monto y guarda, then el movimiento se suma al Ejecutado, se recalculan los roll-ups y aparece en recientes.
Given la grilla de escritorio, when edita una celda Presupuestado/Ejecutado de una hoja, then el valor se fija y los ancestros (categoría/grupo/tipo) se recalculan por roll-up al instante.
Given una categoría con movimientos, when el usuario la borra, then sus movimientos se reasignan a la categoría "Sin asignar" del mismo grupo y no se pierde historial (cero huérfanos).
Given el dashboard, when cambia el filtro Mes/Año, then los 7 indicadores reflejan el periodo seleccionado con la semántica de color correcta.
Given una pantalla pequeña (móvil) en v1, when el usuario abre la app, then ve solo el módulo de registro compacto (no la grilla de presupuesto ni el dashboard).
Given cualquier flujo end-to-end (captura → categorías → presupuesto → dashboard en escritorio), when se recorre completo, then los datos son coherentes entre superficies y no hay bugs. (Criterio de éxito principal de la v1.)

## Hard Constraints

- Stack: Next.js 15, Tailwind CSS v4, shadcn/ui, Recharts, Lucide, TypeScript.
- Tema oscuro único (sin modo claro), tipografía mono (Fira Code), sistema de diseño César Augusto con tokens exactos de la spec.
- Moneda COP, idioma UI español; código e identificadores en inglés.
- Hosting inicial: Next.js en Docker sobre Ultron (Pi 5, 8GB RAM), Nginx como reverse proxy; dejar preparado para hosting profesional.
- El README debe explicar decisiones técnicas, no solo cómo correr el proyecto.

## Out of Scope

**v1 (MVP) — fuera de alcance, con andamiaje preparado donde se indica:**
- Presupuesto y Dashboard en móvil — en v1 el móvil (responsive compacto) muestra SOLO el módulo de registro; presupuesto/dashboard son de escritorio (→ Fase 2 puede ampliar la experiencia móvil).
- Distribución proporcional al editar un nodo padre — ELIMINADO del alcance; en la grilla se editan las hojas y los padres se recalculan por roll-up (los padres no se editan directamente para repartir).
- Multiusuario real / login funcional / libros compartidos — solo se deja el ANDAMIAJE de datos (→ Fase 2).
- APIs externas (conexiones para registros/consultas) — solo se deja el ANDAMIAJE (→ Fase 2).
- Multi-año y multi-moneda — v1 es single-year (2026) y single-currency (COP) (→ Fase 2).

**Post-MVP (Fase 5), fuera de v1:**
- Drag-and-drop para reordenar grupos/categorías (D-7).
- Exportación de datos.
- Tweaks (acento/densidad/varianza) como preferencias de usuario — en v1 quedan como valores base fijos (comfortable + subtle + acento acero).

## Decisiones confirmadas (2026-07-01)

- **D-1:** Tipos FIJOS (Gasto/Ingreso/Transferencia como eje de signo; Grupos editables como capa adicional). Opción A.
- **D-2 (borrado):** los movimientos de una categoría borrada se reasignan a una categoría fija "Sin asignar" por grupo (no a una hermana); "Sin asignar" es auto, no editable/borrable y solo visible cuando tiene datos.
- **D-4 (distribución proporcional):** ELIMINADA. Se editan las hojas; los padres se recalculan por roll-up.
- **Plataforma:** app WEB responsive. Móvil v1 = solo módulo de registro (compacto). Escritorio = app completa. Sin teclado numérico ad-hoc.
- **Persistencia v1:** localStorage.
- **Éxito v1:** demo funcional completa end-to-end, datos coherentes, sin bugs.
